# 스마트팩토리 RTD 노코드 룰 빌더 플랫폼 MVP PRD

> **공통 요소 참조**: 인증, User 모델, 공통 기술 스택, 모듈 아키텍처 → [`prd-common.md`](./prd-common.md)
> **독립 실행 가능**: RTD 모듈은 MCS 없이 단독으로 실행할 수 있습니다.

---

## 🎯 핵심 정보

**목적**: 반도체 공정 현장 엔지니어가 개발자 없이 디스패칭 룰을 직접 생성·수정·검증할 수 있도록 SQL 자동 생성 기반 노코드 룰 빌더 플랫폼 제공
**사용자**: 반도체 제조 공정 현장 엔지니어 (비개발자) — 룰 변경 시마다 개발팀에 의존하는 문제 해결

---

## 🔌 모듈 독립 실행

RTD 모듈은 MCS 모듈 없이 독립적으로 실행 가능합니다.

| 실행 모드 | MCS 연동 상태 | 영향받는 기능 |
|----------|-------------|------------|
| **RTD 단독 실행** | 비활성 | MCS-RTD 이벤트 전송 기능만 비활성. 룰 빌더·시뮬레이터·모니터링 전체 기능 정상 동작 |
| **RTD + MCS 통합 실행** | 활성 | 디스패칭 결과가 MCS의 MacroCommand로 자동 전달되고, MCS 반송 완료 시 다음 디스패칭이 자동 트리거됨 |

> 단독 실행 시 룰 시뮬레이터(F007)의 결과를 수동으로 확인하며 룰 효과를 검증합니다.

## 🚶 사용자 여정

```
1. 로그인 페이지 (→ prd-common.md FC001)
   ↓ 로그인 성공

2. 대시보드 (홈)
   ↓ 룰 관리 시작 클릭

3. 룰 그룹 관리 페이지
   ↓ 룰 그룹 선택 → 플로우 빌더 이동
   ↓ 장비-이벤트 매핑 설정 클릭

   [매핑 설정] → 장비-이벤트-룰그룹 매핑 설정 (동일 페이지 패널)
   ↓
   [룰 플로우 빌더 이동]

4. 룰 플로우 빌더 페이지
   ↓ 시퀀스 블록 추가/편집

   [블록 클릭] → 우측 노드 설정 사이드 패널 열기 → 테이블·조건 설정 → SQL 자동 생성
   [블록 속성 편집] → isMandatory, filterSequence, jumpNextSequence 설정
   [자연어 입력 — F011, Phase 5] → LLM이 룰 블록 시퀀스 자동 생성 → 캔버스 렌더링 → 사용자 수정
   ↓ 저장 완료

5. 룰 시뮬레이터 페이지
   ↓ 테스트 실행 → 시퀀스별 중간 결과 확인

   [검증 성공] → 즉시 적용 → 대시보드로 이동
   [검증 실패] → 오류 내용 표시 → 플로우 빌더로 돌아가 수정
   ↓

6. 모니터링 대시보드 페이지
   ↓ 실시간 실행 로그 및 성능 통계 확인

7. 완료 → [룰 추가 생성] 또는 [모니터링 계속 확인]
```

## ⚡ 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|-------------|------------|
| **F001** | 룰 그룹 관리 | 룰 그룹 트리 구조 CRUD + Fallback 계층 시각화 (장비전용→EQP_FULL/EQP_EMPTY→EQP_COMMON) | 룰의 최상위 단위, 이 없이는 룰 체계 자체가 없음 | 룰 그룹 관리 페이지 |
| **F002** | 장비-이벤트-룰그룹 매핑 설정 | 장비(ruleObjectId)+이벤트(ruleEventId)를 룰그룹에 연결하는 설정 패널 | 어떤 장비·이벤트에 어떤 룰그룹을 적용할지 지정 | 룰 그룹 관리 페이지 |
| **F003** | 룰 플로우 빌더 | 드래그앤드롭으로 시퀀스 블록(DmsRuleRelationVO) 추가·편집·순서 변경. 각 블록에서 룰 타입(Data/SubData/Filter/Join/Groupby/Sort/Method), isMandatory(Y/N/O), filterSequence, jumpNextSequence 편집 | 노코드의 핵심 화면 — 룰 실행 흐름을 시각적으로 구성 | 룰 플로우 빌더 페이지 |
| **F004** | 쿼리 빌더 (노드 사이드 패널) | 노드 클릭 시 우측 사이드 패널에서 기준 테이블 체크박스 선택, 문장형 조건(컬럼/연산자/값) 추가, MES 요청 파라미터·이전 룰 결과 바인딩 → SQL 자동 생성 → DmsRuleQueryVO.ruleQueryString 저장 | 개발자 없이 엔지니어가 SQL 작성하는 핵심 노코드 기능. SQL 직접 노출 없이 문장형 UI 제공 | 룰 플로우 빌더 페이지 |
| **F005** | 정렬 조건 편집기 | 컬럼 선택 + ASC/DESC + 가중치/백분율 설정 → DmsRuleSortVO 저장 | Sort 타입 룰에 필수, 가중치 기반 정렬 지원 | 룰 플로우 빌더 페이지 |
| **F006** | 파라미터 바인딩 편집기 | 쿼리 바인딩 파라미터 키-값 편집 테이블 → DmsRuleQueryParamVO 저장 | 쿼리 실행 시 동적 변수 바인딩에 필수 | 룰 플로우 빌더 페이지 |
| **F007** | 룰 시뮬레이터 | 저장 전 실제 DB 데이터로 테스트 실행, 시퀀스별 중간 결과 건수 미리보기, 룰 유효성 검증 (순환 참조, 잘못된 filterSequence) | 잘못된 룰 배포 방지 — 현장 적용 전 반드시 검증 필요 | 룰 시뮬레이터 페이지 |
| **F008** | 룰 실행 로그 모니터링 | DmsRuleRunningResultVO 기반 — LotID, Sequence, RuleType, 결과건수, 적용여부, 소요시간 실시간 표시 (WebSocket/SSE). **공장 레이아웃·캐리어 이동 모니터링은 MCS 영역** | 룰이 실제로 동작하는지 즉시 확인하는 운영 필수 기능 | 모니터링 대시보드 페이지, 대시보드 |
| **F009** | 룰 실행 통계 및 성능 분석 | RuleClassType별 평균 소요시간·히트율 집계, 기간/룰그룹/장비별 이력 검색 | 룰 효과 측정 및 룰 생성 시간 50% 단축 목표 달성 검증 | 모니터링 대시보드 페이지, 대시보드 |

### 2. MVP 필수 지원 기능

> 인증 기능은 공통(FC001, `prd-common.md`)으로 위임합니다.

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|-------------|------------|
| **F010** | MCS-RTD 이벤트 전송 | 디스패칭 결과를 MCS API로 전송, MCS 반송 완료 이벤트 수신. **단독 실행 시 비활성** | RTD-MCS 통합 연구 요구사항 | 모니터링 대시보드 페이지 |

### 3. 연구 확장 기능 (LLM 프로토타입)

> 논문 실험용으로 가능성 검증 수준의 프로토타입 구현. 전체 자동화가 아닌 few-shot 기반 샘플 시나리오 대상.

| ID | 기능명 | 설명 | 연구 목적 | 관련 페이지 |
|----|--------|------|----------|------------|
| **F011** | 자연어 룰 생성 보조 | LLM을 활용해 자연어 입력("긴급 Lot 우선 디스패칭하고 IDLE 설비만 필터")을 룰 블록 시퀀스로 자동 변환하는 프로토타입. few-shot 프롬프트 기반, 샘플 시나리오 3~5개 대상. 생성된 플로우는 기존 빌더에서 수정 가능 | 노코드 룰 빌더의 자연어 인터페이스 확장 가능성 검증. 참고: Xia et al., IEEE ETFA 2025 Best Paper | 룰 플로우 빌더 페이지 |

### 4. MVP 이후 기능 (제외)

- MCS 반송 제어 및 AI 경로 최적화 (별도 프로젝트)
- 프로필 상세 관리 (아바타, 자기소개)
- 알림 설정 및 이메일 발송
- 룰 버전 관리 및 롤백 UI
- 다국어 지원

### 모니터링 대시보드 페이지

> **구현 기능:** `F008`, `F009`, `F010` | **인증:** 로그인 필요
> **범위 주의**: 이 페이지는 **룰 실행 로그** 모니터링에 집중합니다. 공장 레이아웃 기반 물류 모니터링(장비 상태, 캐리어 이동 애니메이션)은 MCS 모듈의 영역입니다.

| 항목 | 내용 |
|------|------|
| **역할** | 디스패칭 룰이 실제로 어떻게 실행되었는지 실시간 로그와 성능 지표를 분석하는 화면 |
| **진입 경로** | 좌측 메뉴 "모니터링 대시보드" 클릭 |
| **사용자 행동** | 실시간 룰 실행 로그 확인, 기간/룰그룹/장비별 이력 검색, 성능 통계 차트 확인 |
| **주요 기능** | • 룰 실행 로그 테이블: LotID, Seq, RuleType, 결과건수, 적용여부, 소요시간 (WebSocket/SSE 연동) (F008)<br>• 기간/룰그룹ID/장비ID 검색 필터 (F008)<br>• RuleClassType별 평균 소요시간 바 차트 (F009)<br>• Filter 룰 히트율 순위 테이블 (어떤 룰이 가장 효과적인지) (F009)<br>• 일별 룰 실행 건수 추이 라인 차트 (F009)<br>• MCS 연동 상태 표시 (통합 실행 시): MCS API 연결 상태, 최근 디스패칭 전송 결과 (F010) |
| **다음 이동** | 특정 룰그룹 클릭 → 룰 플로우 빌더 페이지로 이동 (해당 룰그룹 컨텍스트) |

## 🗄️ 데이터 모델

기존 Dms 엔티티 구조를 **DB 스키마 변경 없이** 그대로 재사용합니다.

### RuleGroup (DMS_RULE_GROUP_DEF)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleGroupId | 룰 그룹 고유 ID | PK |
| ruleGroupName | 룰 그룹 이름 | String |
| ruleGroupType | 그룹 유형 (DISPATCHING, ROUTING) | String |
| isUsable | 사용 여부 | String |
| description | 설명 | String |

### RuleObject (DMS_RULE_OBJECT)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleObjectId | 장비 단위 ID | PK |
| ruleEventId | 이벤트 유형 ID | PK |
| ruleGroupId | 연결된 룰 그룹 | → RuleGroup |
| siteId | 사이트 식별자 | PK |
| isUsable | 사용 여부 | String |

### RuleRelation (DMS_RULE_RELATION)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleGroupId | 속한 룰 그룹 | PK → RuleGroup |
| ruleId | 실행할 룰 | PK → RuleDef |
| sequence | 실행 순서 번호 | PK, Integer |
| isMandatory | 필수 여부 (Y/N/O) | String |
| filterSequence | 이전 시퀀스 참조 번호 | Integer |
| jumpNextSequence | 조건부 점프 목적 시퀀스 | Integer |
| jumpNextSequenceCondition | 점프 조건 (COUNT>0, COUNT=0) | String |
| ruleSortId | 연결된 정렬 ID | → RuleSort |

### RuleDef (DMS_RULE_DEF)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleId | 룰 고유 ID | PK |
| ruleName | 룰 이름 | String |
| ruleClassId | 룰 클래스 분류 | → RuleClass |
| ruleType | 룰 유형 | String |
| ruleCondition | 룰 적용 조건 | String |

### RuleQuery (DMS_RULE_QUERY)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleQueryId | 쿼리 ID | PK |
| ruleQueryVersion | 버전 | PK |
| ruleQueryString | SQL 쿼리 문자열 (쿼리 빌더가 자동 생성) | String (4000자) |
| ruleQueryType | 쿼리 유형 | String |

### RuleSort (DMS_RULE_SORT)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleSortId | 정렬 ID | PK |
| sortColumn | 정렬 컬럼명 | String |
| weightValue | 가중치 | Decimal |
| fromPercent | 적용 범위 시작 백분율 | Decimal |
| toPercent | 적용 범위 끝 백분율 | Decimal |
| orderBy | ASC / DESC | String |

### RuleQueryParam (DMS_RULE_QUERY_PARAM)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| ruleQueryId | 쿼리 ID | PK → RuleQuery |
| ruleQueryVersion | 버전 | PK → RuleQuery |
| paramKey | 바인딩 파라미터 키 | PK, String |
| paramValue | 바인딩 파라미터 값 | String |
| targetColumn | 대상 컬럼명 | String |

### RuleClass (DMS_RULE_CLASS)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|

| ruleClassName | 룰 클래스 이름 | String |
| ruleClassType | 룰 클래스 유형 (F009 통계에서 집계 기준) | String |

### RuleRunningResult (DMS_RULE_RUNNING_RESULT)
| 필드 | 설명 | 타입/관계 |
|------|------|----------|
| uuid | 자동 생성 고유 키 | PK |
| lotId | 처리 대상 Lot ID | String |
| ruleId | 실행된 룰 ID | → RuleDef |
| sequence | 실행 시퀀스 번호 | Integer |
| count | 결과 건수 | Integer |
| startTime / endTime | 실행 시간 | Timestamp |
| isDispatching | 디스패칭 적용 여부 | String |

## 🛠️ 기술 스택

> 공통 스택(`prd-common.md`) 기반에서 RTD 모듈 전용 스택을 추가합니다.

### 🔀 RTD 전용 — 플로우 빌더

- **React Flow 12.x** - 드래그앤드롭 노드 기반 플로우 차트 (룰 시퀀스 빌더 핵심)

### 🤖 RTD 전용 — LLM 연동 (확장)

- **OpenAI API 또는 Anthropic API** - 자연어 → 룰 블록 시퀀스 변환 프로토타입 (few-shot prompting)

### 🔌 RTD 전용 — 백엔드 연동

- **Next.js Route Handlers** - 연구 단계 경량 백엔드 (Next.js `app/api/` 엔드포인트)
  - 추후 확장 시 Java/Spring Boot로 교체 가능 (API 계약 동일, base URL만 변경)
- **Supabase** - PostgreSQL 데이터베이스 + Realtime 구독
- **SSE (Server-Sent Events)** - 실시간 모니터링 데이터 수신 (Supabase Realtime 활용)

### 🗄️ RTD 전용 — 데이터 페칭

- **TanStack Query 5.x** (React Query) - 서버 상태 관리 및 캐싱
